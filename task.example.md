# Playwright 一体化任务模板

请基于当前仓库完成一次完整的 Playwright 自动化任务，并严格遵守仓库中的 `AGENTS.md`、现有目录结构、命名规范、分层约束与代码风格。

本模板适用于以下类型的需求：

- `【已有大部分能力，只需补少量 page / flow 并完成用例】`
- `【不确定仓库是否已有能力，需要 AI 先自查再落代码】`
- `【希望一轮完成 page / flow / test / 验证】`

## 任务目标

- 场景名称：`【填写场景名】`
- 用例类型：`【smoke / e2e / regression】`
- 业务目标：`【填写最终要验证的业务结果】`

## 执行模式要求

在开始编码前，请按以下顺序执行，不要跳步：

1. 阅读并遵守 `AGENTS.md`
2. 检查仓库中是否已有可复用的：
   - pages
   - flows
   - fixtures
   - helpers / utils
   - tests
3. 判断当前任务属于以下哪类：
   - `【已有能力足够，直接补或修改用例】`
   - `【已有能力不够，需要最小补充 page / flow】`
   - `【当前页面结构复杂，需先补解析或页面能力，再完成用例】`
4. 优先复用已有能力，不重复创建同类对象
5. 仅补充本次场景所需的最小必要能力
6. 最终必须运行相关 Playwright 用例，并基于真实结果汇报

## 实现约束

- 使用 `Playwright Test`
- `describe` / `test` / `test.step` 使用中文命名
- 页面动作放在 `pages`
- 业务编排放在 `flows`
- 测试文件只保留测试意图与断言，不堆砌底层操作细节
- 优先使用 `data-testid`
- 禁止使用 `waitForTimeout`
- 禁止通过直跳业务内页 URL 绕过正常入口流程，除非本次任务明确是纯页面契约解析且用户已指定页面上下文
- 优先复用已有 page object；如需补充，请保持 pages / flows 职责清晰
- 选择器要稳定，避免过度依赖文案、层级、nth()
- 如定位不稳定，应优先补充 page object / flow，而不是把复杂 locator 直接堆在 spec 中
- page 不要写业务选择策略
- flow 表达业务意图，负责顺序、分支、兜底与状态恢复
- 如页面字段可能为空，返回结构中应明确表达可空语义，而不是抛出无意义异常
- 如读取结果为动态数据，请返回结构化快照对象，便于测试做对比校验

## 代码产出要求

请根据实际需要，直接修改或新增以下内容：

- `pages/...`
- `flows/...`
- `tests/...`

并确保：

1. 新增代码可运行
2. 命名符合仓库风格
3. 不引入无关重构
4. 仅实现本次场景所需的最小必要改动

## 输出要求

完成后请输出：

1. 先说明本次判断结果：
   - `【直接复用已有能力】`
   - `【补充了最小 page / flow 能力】`
   - `【同时补了 page / flow / test】`
2. 本次新增或修改了哪些文件
3. 核心实现说明
4. 执行命令
5. 实际验证结果
6. 如未能完成，请明确说明卡点与原因，不要伪造完成状态

补充要求：

- 必须运行相关 Playwright 用例，并基于真实执行结果汇报
- 不允许仅根据代码推断“理论上应当通过”
- 如果只完成了代码修改但未完成验证，必须明确说明未验证部分与原因
