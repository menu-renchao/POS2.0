# POS UI 自动化框架说明

## 1. 文档目标

本文档用于向团队成员介绍当前 `pos2.0` 自动化项目的整体设计、模块职责、交互逻辑和扩展方式。

这个项目不是一次性的脚本集合，而是一个基于 Playwright Test + TypeScript 的可维护自动化框架，目标是：

- 用稳定、可复用的 Page Object 和 Flow 表达页面能力与业务意图
- 用统一的测试分层承载 smoke、业务回归、页面契约校验
- 用中文步骤和 Allure 报告提升执行可读性
- 用清晰边界保证后续可以持续交给 Codex 或人工维护

---

## 2. 技术栈与运行方式

### 2.1 核心技术

- 测试运行器：`@playwright/test`
- 语言：`TypeScript`
- 报告：`allure-playwright` + `allure-commandline`
- 运行浏览器：`chrome`

### 2.2 关键脚本

`package.json` 中已定义以下常用命令：

- `npm test`：运行全部测试
- `npm run test:smoke`：仅运行 smoke 测试
- `npm run test:e2e`：仅运行 e2e 测试
- `npm run test:headed`：以 headed 模式运行
- `npm run codegen`：打开 Playwright Codegen
- `npm run allure:generate`：生成 Allure 报告
- `npm run allure:open`：打开 Allure 报告

### 2.3 运行配置

主配置文件是 `playwright.config.ts`，主要职责如下：

- 指定测试目录为 `tests/`
- 配置全局超时和断言超时
- 配置 Allure reporter
- 指定 `tests/setup/global.setup.ts` 为全局初始化入口
- 统一设置 `baseURL`
- 失败时自动截图，重试时保留 trace
- 使用 `chrome` 项目执行

## 3. 项目目录结构

```text
pos2.0
├─ docs/                      文档
├─ fixtures/                  Playwright fixture 扩展
├─ flows/                     业务流程封装
├─ pages/                     页面对象层
├─ test-data/                 环境数据、选项枚举、测试输入
├─ tests/
│  ├─ e2e/                    业务回归 / 页面契约
│  ├─ setup/                  全局初始化
│  └─ smoke/                  稳定性冒烟
├─ utils/                     纯工具与公共辅助
├─ AGENTS.md                  项目自动化约束
├─ package.json
└─ playwright.config.ts
```

这套结构的核心思想是：

- `pages/` 只描述“页面能做什么、能读到什么”
- `flows/` 只描述“业务想完成什么”
- `tests/` 只描述“验证什么结果”

## 4. 框架分层设计

## 4.1 `pages/` 页面对象层

`pages/` 是整个项目最核心的一层。它只封装页面结构、定位器、页面动作和页面读取。

这一层不能写业务策略，例如：

- 不能写“选择任意可用台”
- 不能写“优先选某种订单”
- 不能写“先失败再换备选方案”

这些都应该放到 `flows/`。

### 当前页面模块

#### `home.page.ts`

职责：

- 打开主页
- 等待主页加载完成
- 校验主页关键入口是否可见
- 点击首页功能入口

当前重点支持的入口：

- `Dine In`
- `Delivery`
- `Pick Up`
- `Report`
- `Admin`
- `Recall`

说明：

- POS 首页内容运行在 `#newLoginContainer iframe` 内，所以 `HomePage` 同时承担了 iframe 进入点的作用。
- 所有后续页面流转，都是从首页 UI 入口点击进入，不允许直接拼接内部 hash 地址进入。

#### `license-selection.page.ts`

职责：

- 判断授权选择页面是否出现
- 校验授权列表是否展示
- 选择某种授权类型的可用项
- 点击进入系统
- 校验授权状态

说明：

- 该页面不是每次必出现，因此通常由 flow 决定“如果出现则处理”。

#### `employee-login.page.ts`

职责：

- 判断员工口令输入区域是否出现
- 输入员工密码
- 点击确认

说明：

- 该项目把“登录”理解为员工上下文进入，而不是传统 SaaS 的全局会话登录。

#### `select-table.page.ts`

职责：

- 等待选台页面加载
- 切换区域
- 获取当前区域下可用台列表
- 按桌号查找台位
- 点击桌台
- 读取桌号

说明：

- 这一层只负责“查台、点台、读台”，不负责决定选哪张台。

#### `guest-count-dialog.page.ts`

职责：

- 校验人数弹窗出现
- 选择就餐人数

说明：

- 该页面是 Dine In 进入点菜页前的过渡弹窗。

#### `order-dishes.page.ts`

职责：

- 等待点菜页加载
- 校验桌号、人数等基础上下文
- 点击菜品
- 修改数量
- 处理称重菜、开价菜、规格菜、套餐菜的页面交互
- 校验购物车内容
- 保存订单

说明：

- 该页是页面动作最复杂的模块之一。
- 它只提供低层交互，例如“输入重量”“确认规格弹窗”“选择套餐项”，不决定业务上应该点哪种菜。

#### `recall.page.ts`

职责：

- 等待 Recall 页面加载
- 执行各种筛选条件操作
- 处理人工搜索弹窗
- 打开指定订单详情弹窗
- 读取订单详情快照
- 关闭订单详情弹窗

当前订单详情读取能力包括：

- 客户/会员信息
- 支付状态
- 菜品详情
- 价格汇总
- 订单上下文
- 支付记录

说明：

- 订单详情读取是近期重点建设模块。
- 当前实现已支持在读取完成后关闭弹窗，不污染当前页面状态。
- 定位策略优先使用 `data-testid` 或 `data-test-id`，没有稳定测试属性时再退回语义定位或结构兜底。

#### `admin.page.ts` / `delivery.page.ts` / `pick-up.page.ts` / `report.page.ts`

职责：

- 当前仅保留最小页面对象壳层

说明：

- 这些页面目前还没有形成完整自动化能力。
- 它们的存在主要是为后续扩展预留统一目录和对象命名。

## 4.2 `flows/` 业务流程层

`flows/` 用来表达多步业务意图。它组合多个 page 方法，决定顺序、选择策略、回退逻辑和最终返回结果。

简单理解：

- `pages/` 回答“这个页面能做什么”
- `flows/` 回答“为了完成一个业务目标，应该怎么串起来”

### 当前 flow 模块

#### `home.flow.ts`

职责：

- 打开首页
- 等待首页加载成功

这是所有业务流程的公共入口。

#### `license-selection.flow.ts`

职责：

- 当授权选择页出现时，进入可用授权

这是一个典型的“页面是否出现由场景决定，但业务希望统一处理”的 flow。

#### `employee-login.flow.ts`

职责：

- 使用员工密码进入系统

它把口令输入和确认动作组合成一个更贴近业务的话术。

#### `select-table.flow.ts`

职责：

- 选择当前区域任意可用台
- 选择指定区域的指定桌号
- 在人数弹窗中选择人数并进入点菜页

这是页面边界和业务边界区分得最清楚的一层：

- 页面负责“读出有哪些台”
- flow 负责“选第一个可用台”或“按区域和桌号选择”

#### `order-dishes.flow.ts`

职责：

- 添加普通菜
- 添加第一道可用菜
- 添加称重菜
- 添加套餐菜
- 添加规格菜
- 添加开价菜
- 统一按参数分发点菜逻辑

说明：

- 它负责业务判断，例如当 `weight` 有值时走称重菜流程，当 `comboSelections` 有值时走套餐流程。
- 这正是业务策略该存在的位置。

#### `recall.flow.ts`

职责：

- 按组合条件搜索订单
- 清空搜索条件
- 查看指定订单详情

其中“查看指定订单详情”的 flow 逻辑是：

1. 确认 Recall 页面已加载
2. 根据订单号打开订单详情弹窗
3. 读取完整订单详情快照
4. 在 `finally` 中关闭弹窗

这个设计保证了：

- 流程表达业务意图
- 页面只负责打开、读取、关闭
- 即使读取中出现异常，也尽量恢复 UI 状态

## 4.3 `fixtures/` Playwright 夹具层

当前文件：`fixtures/test.fixture.ts`

职责：

- 在 Playwright `test` 上扩展项目自定义 fixture
- 为测试统一注入页面对象实例

当前注入的 fixture 包括：

- `homePage`
- `licenseSelectionPage`
- `employeeLoginPage`

价值：

- 测试体不需要自己 new 页面对象
- 页面对象生命周期统一交给 fixture 管理
- 测试写法更简洁

后续如果新增页面对象，通常也应从这里统一注入。

---

## 4.4 `test-data/` 测试数据层

### `env.ts`

职责：

- 管理环境配置
- 提供系统基础访问地址、主页路径、标题等数据

当前组合逻辑：

- `baseURL` 默认指向 `http://192.168.0.89:22080`
- `homePath` 为 `/kpos/front2/myhome.html`

这使得配置层和测试层解耦，避免在测试里硬编码地址。

### `recall-search-options.ts`

职责：

- 定义 Recall 搜索所需的选项类型和枚举值

包括：

- 支付状态
- 订单状态
- 订单类型
- 支付方式
- 产品线
- 人工搜索标签

价值：

- 让 Recall 搜索的参数表达更结构化
- 避免在 flow 和 test 中散落魔法字符串

## 4.5 `utils/` 工具层

### `step.ts`

职责：

- 提供 `@step(...)` 装饰器
- 自动把 page/flow 方法包装成 Playwright `test.step`

作用：

- 在报告中显示中文步骤
- 让页面动作、业务动作都能被结构化追踪
- 降低测试正文中的步骤样板代码

这是本框架可读性最重要的基础设施之一。

### `expectations.ts`

职责：

- 放置可复用的断言辅助方法

当前已有：

- `expectPathname`

这类工具适合承载“可复用但不属于单个页面”的断言逻辑。

## 4.6 `tests/` 测试层

`tests/` 不是直接堆脚本，而是按测试目标分层。

### `tests/setup/`

#### `global.setup.ts`

职责：

- 在测试运行前校验配置是否可用

当前主要是验证 `baseURL`，确保基础运行环境具备执行条件。

### `tests/smoke/`

职责：

- 验证主流程和关键入口是否可用
- 关注稳定性信号，不做过深业务校验

当前 smoke 覆盖包括：

- 首页打开
- 授权选择
- 授权 + 员工登录
- Dine In 导航
- Dine In 到点菜页
- 点菜页基本能力
- Recall 搜索
- Recall 订单详情读取
- `@step` 装饰器能力

适用场景：

- 发版前快速验主链
- 环境冒烟
- 核心入口可达性确认

### `tests/e2e/`

职责：

- 表达更完整的业务意图
- 覆盖页面契约、业务参数分支和易漂移区域

当前 e2e 覆盖包括：

- 点菜数量修改
- 套餐菜数量逻辑
- Recall 页面关键选择器契约
- Recall 订单详情解析契约

适用场景：

- 功能回归
- 页面结构漂移防护
- 新需求开发后的定向验证

## 5. 主流程交互逻辑

下面是当前项目中最重要的几条交互链路。

## 5.1 进入系统主链路

```text
打开首页
  -> 如出现授权选择页则进入可用授权
  -> 输入员工口令
  -> 进入 POS 首页主功能区
```

对应层次分工：

- `HomeFlow.openHome()`：负责打开首页
- `LicenseSelectionFlow.enterWithAvailableLicense()`：负责授权处理
- `EmployeeLoginFlow.enterWithEmployeePassword()`：负责员工上下文进入

这条链路是大多数业务测试的前置步骤。

## 5.2 Dine In 点菜链路

```text
首页点击 Dine In
  -> 进入选台页
  -> 选择桌台
  -> 弹出人数对话框
  -> 选择人数
  -> 进入点菜页
  -> 点菜 / 改数量 / 处理规格、称重、套餐、开价
  -> 校验购物车或保存订单
```

对应模块关系：

- `home.page.ts`：点击 `Dine In`
- `select-table.page.ts`：页面级选台能力
- `select-table.flow.ts`：决定选哪张台、选几位客人
- `order-dishes.page.ts`：页面级点菜动作
- `order-dishes.flow.ts`：决定点菜策略

### 点菜流程的参数分发逻辑

`OrderDishesFlow.addDishToCart()` 是一个统一入口，它根据参数自动分流：

- 有 `weight`：走称重菜流程
- 有 `price`：走开价菜流程
- 有 `specifications`：走规格菜流程
- 有 `comboSelections`：走套餐菜流程
- 其他情况：走普通菜流程

这样做的好处是：

- 测试调用方式统一
- 业务变化集中在 flow 层
- 页面层保持纯页面动作

## 5.3 Recall 搜索链路

```text
首页点击 Recall
  -> 进入 Recall 页面
  -> 根据需要清空现有筛选
  -> 设置支付状态 / 订单状态 / 订单类型 / 支付方式 / 产品线
  -> 如有人工搜索则打开弹窗并提交
  -> 返回当前 Recall 页面对象
```

对应模块关系：

- `home.page.ts`：点击 `Recall`
- `recall.page.ts`：筛选条件的页面动作
- `recall.flow.ts`：组合筛选逻辑

说明：

- 页面只负责“点哪个筛选器、填哪个输入框”
- flow 才负责“本次应该组合哪些条件”

## 5.4 Recall 订单详情链路

```text
首页点击 Recall
  -> 打开 Recall 页面
  -> 根据订单号点击订单卡片
  -> 打开订单详情弹窗
  -> 读取订单信息
  -> 主动关闭弹窗
```

当前读取内容包括：

- 客户/会员信息，可为空
- 支付状态，可为空
- 菜品明细，可为空
- 价格明细
- 订单类型、桌号、人数、服务员，可为空
- 支付记录，可为空

### 订单详情读取设计要点

1. page 不写业务选择策略  
`RecallPage` 只负责“打开哪张订单的详情”和“如何读取当前弹窗内容”。

2. flow 表达业务意图  
`RecallFlow.viewOrderDetails()` 直接表达“查看指定订单详情并恢复页面状态”。

3. 读取结束必须关闭弹窗  
无论成功还是异常，flow 都在 `finally` 中执行关闭，避免影响后续步骤。

4. 解析对 DOM 漂移要有韧性  
近期已针对 Recall 详情页做过增强：

- 优先识别 `data-testid` / `data-test-id`
- 对 CSS module hash 变化不敏感
- 菜名、价格汇总、座位区、支付信息支持结构兜底
- `3(1)` 这类头部信息会解析为 `tableName = 3`、`guestCount = 1`

## 6. 报告与调试逻辑

## 6.1 中文步骤报告

项目要求：

- `pages/` 和 `flows/` 中的方法都使用中文 `@step(...)`
- `describe`、`test` 标题也统一使用中文

价值：

- Allure 报告更容易让业务、测试、开发共同阅读
- 出问题时能快速定位失败发生在页面动作层还是业务层

## 6.2 当前如何避免“红了但最终通过”的假红灯

项目早期如果直接使用：

- `expect(...).toPass(...)`
- `expect.poll(...)`

虽然最终可能通过，但中间失败历史会出现在 Allure 里，容易让阅读者误以为用例不稳定。

当前已经通过 `utils/wait.ts` 中的 `waitUntil()` 统一承接这类条件轮询：

- 轮询期间不制造中间失败步骤
- 到达成功条件后再做最终断言
- 超时后抛出包含最后值的明确错误

因此现在的推荐实践是：

- 页面和 flow 中需要等待条件变化时，优先使用 `waitUntil()`
- 只把最终结果交给 `expect(...)`

这样报告信号更干净，也更适合同事排查问题。

---

## 7. 定位器策略与编码规范

项目当前约束如下：

1. 优先使用 `data-testid`  
只要页面提供稳定测试属性，优先使用它。

2. 其次使用语义定位  
没有稳定测试属性时，再考虑：

- `getByRole`
- `getByLabel`
- `getByText`

3. 不默认使用脆弱选择器  
避免：

- 长 CSS 链
- `nth-child`
- XPath

4. 如果某个选择器有复用价值，必须集中定义  
不要把原始选择器散落在 page 方法内部。

5. 页面方法必须是页面能力，不要混入业务决策  
例如“选第一张可用台”是 flow 逻辑，不是 page 逻辑。

## 8. 如何扩展这个框架

## 8.1 新增一个页面自动化模块

建议步骤：

1. 在 `pages/` 下创建新页面对象
2. 把稳定定位器集中定义为字段或私有定位器工厂
3. 只实现页面动作与读取
4. 为每个公开方法补中文 `@step(...)`
5. 如测试会频繁用到，在 `fixtures/test.fixture.ts` 中注入 fixture

## 8.2 新增一个业务流程

建议步骤：

1. 在 `flows/` 下创建对应 flow
2. 组合现有 page 方法完成业务目标
3. 把选择策略、容错顺序、意图表达放在 flow 中
4. 返回业务友好的结果对象

## 8.3 新增一条 smoke

适合验证：

- 页面是否能打开
- 关键入口是否存在
- 核心流程是否能走通

不适合在 smoke 里做：

- 大量细节断言
- 复杂分支覆盖
- 对非稳定 UI 文本的深度依赖

## 8.4 新增一条 e2e / 契约测试

适合验证：

- 参数分支
- 业务规则
- 易变选择器
- 解析逻辑

例如 Recall 最近的详情解析增强，就是通过 e2e 契约测试来锁住结构漂移风险。

## 10. 一句话理解这套框架

这不是“直接操作浏览器的脚本集合”，而是一个把 POS 页面能力、业务流程和测试验证分层组织起来的自动化工程：

- `pages/` 负责页面能力
- `flows/` 负责业务意图
- `tests/` 负责结果验证
- `fixtures/` 负责注入
- `test-data/` 负责数据
- `utils/` 负责公共基础设施

团队后续只要继续遵守这套边界，项目就能持续扩展，而不会退化成难维护的大脚本。
