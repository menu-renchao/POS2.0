# POS 自动化项目 AI 协作指南

## 1. 文档目标

本文档用于指导团队在 `pos2.0` 项目中使用 AI 进行需求分析、测试设计、代码生成、代码审查和问题排查。

它不是替代 `AGENTS.md` 的规则文件，而是解释如何把这些规则转化为稳定的 AI 协作流程。所有具体实现仍必须以 `AGENTS.md` 为最高项目约束。

本项目中的 AI 协作目标是：

- 让 AI 先理解项目结构，再做最小必要改动
- 让 AI 输出符合 Playwright Test + TypeScript 工程规范的代码
- 让 page、flow、test、test-data、utils 的边界持续清晰
- 让每次交付都能追溯到需求、设计、实现和验证结果

一句话原则：

> 先把需求讲清楚，再让 AI 执行；先让 AI 读懂项目，再让 AI 改代码；先验证真实结果，再宣布完成。

---

## 2. AI 在本项目中的角色

AI 在本项目中不是“录制脚本工具”，而是自动化工程协作者。

AI 适合承担：

- 根据需求拆分 page、flow、test 的职责
- 基于现有模式新增测试用例
- 补充页面对象、业务流程和测试数据
- 修复选择器漂移、等待策略和报告步骤问题
- 整理测试计划、迁移计划和验证报告
- 对变更做代码审查或风险分析

AI 不应承担：

- 在没有页面信号时盲写选择器
- 绕过真实 POS 入口直接打开内部页面
- 把业务策略塞进 page object
- 在没有运行验证时声称测试通过
- 为了让用例通过而引入宽泛兜底选择器
- 对不明确需求做大范围重构

日志检索类、代码审查类任务，应按项目要求使用 `GPT-5.3-Codex-Spark` 模型，并通过 subagent 调用。

## 3. 给 AI 的 Prompt 标准结构

参考通用 Prompt 七段式，本项目建议每个复杂任务都按以下结构给 AI：

### 3.1 角色设定

说明 AI 应以什么身份工作。

示例：

```text
你是一个熟悉 Playwright Test、TypeScript、Page Object Model 和 POS 业务自动化的资深测试开发工程师。
你必须优先遵守当前仓库 AGENTS.md，并保持 page、flow、test 分层清晰。
```

### 3.2 任务说明

明确要完成什么，不要只写“帮我加个用例”。

示例：

```text
请为 Recall 页面新增一个 e2e 用例，验证通过订单号打开订单详情后可以读取菜品明细和价格汇总。
```

### 3.3 背景信息

提供入口、业务背景、已知约束和相关文件。

示例：

```text
必须从 POS 首页进入，不允许直接打开 Recall hash。
相关文件：
- pages/recall.page.ts
- flows/recall.flow.ts
- tests/e2e/recall-order-details.spec.ts
```

### 3.4 具体要求

把任务拆成可执行步骤。

示例：

```text
请按以下步骤处理：
1. 先读取 AGENTS.md 和相关已有实现。
2. 判断是否需要新增 page 能力、flow 能力或只新增 test。
3. 优先复用已有 locator、fixture、flow 和 test-data。
4. 新增公共 page/flow 方法时必须添加中文 @step(...)。
5. 完成后运行相关 Playwright 用例或说明无法运行的原因。
```

### 3.5 输出格式

明确 AI 最后应如何汇报。

示例：

```text
完成后请输出：
1. 修改文件列表。
2. 分层设计说明。
3. 运行命令和真实结果。
4. 未验证项和原因。
```

### 3.6 示例说明

如果任务有相似实现，应告诉 AI 参考哪一个。

示例：

```text
请参考 tests/e2e/recall-order-details.spec.ts 的测试结构，
以及 flows/recall.flow.ts 中查看订单详情后恢复弹窗状态的写法。
```

### 3.7 注意事项

把最容易犯错的点显式写出来。

示例：

```text
注意：
- 不要在 spec 中直接写 locator。
- 不要使用 waitForTimeout。
- 不要使用 .or() 堆叠多套选择器。
- 如果页面缺少稳定选择器，应说明需要补 data-testid。
```

## 4. 推荐 AI 开发流程

本项目推荐采用轻量 Spec 驱动流程：

```text
需求输入
  -> 上下文读取
  -> 分层设计
  -> 最小实现
  -> 真实验证
  -> 交付说明
```

### 4.1 需求输入

需求方至少应提供：

- 场景名称
- 用例类型：smoke / e2e / 契约测试 / 维护任务
- 入口和前置条件
- 业务步骤
- 断言要求
- 测试数据
- Jira issue key
- 本次明确不做的范围

### 4.2 上下文读取

AI 修改代码前必须先读取：

- `AGENTS.md`
- `package.json`
- `playwright.config.ts`
- 相关 `pages/`
- 相关 `flows/`
- 相关 `tests/`
- 相关 `test-data/`
- 必要时读取 `utils/step.ts`、`utils/wait.ts`

目标是判断：

- 现有能力是否可复用
- 新增能力应放在哪一层
- 是否存在类似测试或工具方法
- 是否需要先请求前端补稳定 `data-testid`

### 4.3 分层设计

AI 在动手前应先说明：

- page 层新增或复用哪些页面能力
- flow 层新增或复用哪些业务编排
- test 层验证什么结果
- test-data 是否需要新增样本或工厂函数
- utils 是否需要新增纯工具

如果改动很小，设计可以很短，但仍应先判断边界。

### 4.4 最小实现

实现时遵守以下原则：

- 优先修改最少文件
- 优先复用已有 page、flow、fixture、utils
- 不做无关重构
- 不扩大选择器范围
- 不把临时调试逻辑留在代码里
- 不为单个用例写不可复用的大脚本

### 4.5 真实验证

AI 完成后应运行与改动匹配的命令，例如：

```bash
npx tsc --noEmit
npx playwright test tests/smoke/xxx.spec.ts --reporter=line
npx playwright test tests/e2e/xxx.spec.ts --reporter=line
npm run test:smoke
```

如果环境不可达、浏览器不可用或依赖缺失，必须如实说明，不能声称通过。

### 4.6 交付说明

交付时应包含：

- 修改了哪些文件
- 为什么这些逻辑放在对应层
- 运行了哪些命令
- 验证是否通过
- 剩余风险或未验证项

## 5. 本项目分层规则

### 5.1 Page 层

目录：`pages/`

只放页面结构、页面定位器、页面动作和页面读取。

可以做：

- 点击按钮
- 填写输入框
- 打开或关闭弹窗
- 读取表格、卡片、价格、订单详情
- 等待当前页面加载完成
- 跨页面动作返回目标 page object

不能做：

- 选择任意可用台
- 选择第一条可用订单
- 根据业务参数决定走哪条流程
- 失败后换业务方案重试
- 为了通过测试遍历一堆候选选择器

所有稳定定位器应集中定义在 page object 中，或定义为私有 locator factory。

### 5.2 Flow 层

目录：`flows/`

只放业务意图、多步编排、选择策略和状态恢复。

可以做：

- 从首页进入某个业务模块
- 如出现授权页则进入可用授权
- 根据参数选择普通菜、称重菜、套餐菜、规格菜流程
- 查找订单并读取详情
- 在异常时关闭弹窗恢复页面状态

不能做：

- 重新定义 page locator
- 绕过 page 方法直接操作 DOM
- 把低层页面细节复制到 flow

### 5.3 Test 层

目录：`tests/`

只描述验证目标、业务路径和断言。

要求：

- `describe` 和 `test` 标题使用中文
- test-case-level report steps 使用中文
- 元数据使用 Playwright 原生 `test(title, details, body)` 风格
- Jira 链接放在 `details.annotation`
- 不在 spec 中堆底层 locator
- 不在 spec 中重复解析金额等 page 层已能返回的结构

### 5.4 Test Data 层

目录：`test-data/`

适合放：

- 可复用业务样本
- 菜品、选项、客户、库存等测试输入
- 类型稳定的 case matrix
- 动态值工厂函数

不建议在 spec 中直接写：

- `Date.now()` 生成样本
- 多处复用的菜品名称
- 大段业务测试矩阵

### 5.5 Utils 层

目录：`utils/`

适合放纯工具和跨页面基础能力，例如：

- `@step(...)`
- `waitUntil()`
- Jira annotation 构造
- 可复用断言辅助

不要把页面选择器、业务策略或测试数据塞进 `utils/`。

## 6. 定位器与等待策略

AI 生成或修改页面交互时必须遵守以下顺序。

定位器优先级：

1. `data-testid`
2. 稳定语义定位：`getByRole`、`getByLabel`、`getByText`
3. 页面已有稳定结构

禁止默认使用：

- XPath
- 长 CSS 链
- `nth-child`
- 多语言 fallback selector
- `.or()` 堆叠大量候选定位器
- 遍历候选选择器猜测目标元素

等待策略：

- 不使用 `waitForTimeout`
- 需要轮询时优先使用 `utils/wait.ts` 的 `waitUntil()`
- page、flow、helper、test 中避免用 `expect(...).toPass()` 或 `expect.poll()` 制造报告中间失败噪声
- 修改 input 后立即提交的动作，应在确认按钮点击前等待至少 `200ms` 让输入状态稳定

## 7. Spec 驱动在本项目中的落地

复杂任务建议拆成三类文档或说明。

### 7.1 PRD 级说明

适合描述业务目标。

应包含：

- 用户要完成什么
- 为什么要做
- 涉及哪些 POS 页面
- 入口和前置状态
- 成功标准
- 不做范围

### 7.2 Design 级说明

适合描述自动化分层方案。

应包含：

- page 层能力设计
- flow 层业务编排
- test 层断言范围
- test-data 设计
- 异常和状态恢复
- 选择器策略

### 7.3 Task 级说明

适合拆成可执行任务。

应包含：

- 修改文件
- 任务描述
- 验收标准
- 依赖关系
- 验证命令

小任务可以不单独建 PRD/Design/Tasks 文件，但 AI 至少应在回复中讲清楚这些内容。

### 8.1 新增测试用例

```text
请基于当前仓库新增一个 Playwright 用例，并严格遵守 AGENTS.md。

场景名称：
用例类型：smoke / e2e
业务目标：
入口与前置条件：
测试数据：
业务步骤：
断言要求：
Jira：
不做范围：

实现要求：
1. 必须先读取 AGENTS.md 和相关已有 page/flow/test。
2. 优先复用已有能力。
3. 不允许在 spec 中写底层 locator。
4. 新增 page/flow 方法必须使用中文 @step(...)。
5. 不允许使用 waitForTimeout。
6. 完成后运行相关测试并汇报真实结果。
```

### 8.2 补充 Page / Flow 能力

```text
请为当前项目补充可复用的 page / flow 能力。

目标页面：

已有相关文件：
需要新增的页面动作：
需要新增的业务编排：

要求：
1. page 只包含页面动作和读取。
2. flow 只包含业务意图和多步编排。
3. locator 集中定义，不要散落在方法中。
4. 公共方法必须有中文 @step(...)。
5. 如缺少稳定 selector，请明确说明需要补 data-testid。
```

### 8.3 修复失败用例

```text

失败命令：

要求：
1. 先定位失败发生在 page、flow、test、数据还是环境。
2. 不要直接扩大 selector 范围。
3. 不要用 waitForTimeout 掩盖问题。
4. 修复后重跑失败用例。
5. 如判断为产品缺陷，请说明证据，不要盲目改测试。
```

1. page / flow / test 分层是否清晰。
2. locator 是否稳定。
3. 是否存在 waitForTimeout 或报告噪声。
4. 中文 step 和测试标题是否完整。
5. 是否有未验证或伪验证风险。

## 9. AI 产物评审清单

### 9.1 需求理解

- 是否读取 `AGENTS.md`
- 是否检查已有实现
- 是否明确本次范围
- 是否避免无关重构
- 是否说明无法确定的信息

### 9.2 Page 层

- locator 是否集中定义
- 是否优先使用 `data-testid`
- 是否避免脆弱 CSS、XPath、`nth-child`
- 是否没有业务选择策略
- 公共方法是否有中文 `@step(...)`

### 9.3 Flow 层

- 是否表达业务意图
- 是否复用 page 方法
- 是否没有重复定义 locator
- 是否把选择策略放在 flow
- 异常时是否恢复 UI 状态

### 9.4 Test 层

- `describe` 和 `test` 标题是否为中文
- metadata 是否使用 Playwright 原生风格
- Jira issue 是否保留在 spec 文件中
- 测试正文是否业务可读
- 是否避免底层 locator 和重复解析逻辑

### 9.5 验证

- 是否运行相关命令
- 是否报告真实结果
- 是否说明失败原因
- 是否说明未验证项
- 是否没有伪造通过结果

## 10. 常见反模式

### 10.1 直接把操作写进 spec

问题：

- 选择器散落
- 业务意图不清楚
- 无法复用
- 报告不可读

正确方向：

- 页面动作放 `pages/`
- 业务编排放 `flows/`
- spec 只保留业务路径和断言

### 10.2 Page 层写业务策略

问题示例：

```ts
async selectAnyAvailableTable() {
  // 在 page 中遍历区域并选择第一张空桌
}
```

正确方向：

- page 提供 `readAreaNames()`、`selectArea()`、`getAvailableTables()`、`clickTable()`
- flow 实现 `selectAnyAvailableTable()`

### 10.3 为通过用例堆选择器 fallback

问题：

- 看似稳定，实则掩盖 DOM 契约缺失
- 后续失败难以定位
- 违反项目对真实 DOM contract 的要求

正确方向：

- 使用页面真实暴露的稳定 selector
- 缺少 selector 时补或请求补 `data-testid`

### 10.4 直接打开内部页面

问题：

- 绕过授权选择和员工上下文
- 不符合真实 POS 业务路径
- 让测试结果失真

正确方向：

- 统一从 `http://192.168.0.72:22080/kpos/front2/myhome.html` 进入
- 通过 UI flow 导航到目标页面

## 11. 与 AI Native 思想的关系

参考文档中的 AI Native 强调“意图优先、上下文感知、工具化思维、渐进式增强”。

在本自动化项目中，不需要把 AI Native 架构落成产品代码，但可以借用这些思想改进协作方式：

- 意图优先：测试需求先表达业务目标，而不是点击步骤
- 上下文感知：AI 改代码前必须读取项目规则和相似实现
- 工具化思维：把页面能力沉淀为 page，把业务能力沉淀为 flow，把公共能力沉淀为 utils
- 渐进式增强：先补稳定 smoke，再补 e2e 和契约测试，不一次性铺过大范围

AI 完成任务后，推荐按以下格式回复：

```text
完成内容：
- 修改了 ...
- 新增了 ...

分层说明：
- page 层 ...
- flow 层 ...
- test 层 ...

验证：
- 已运行：...
- 结果：...

风险：
- ...
```

如果没有完成，应明确说明阻塞点：

```text
未完成原因：
- ...

已确认信息：
- ...

建议下一步：
- ...
```

## 13. 总结

这个仓库的 AI 协作核心不是“让 AI 多写代码”，而是让 AI 在清晰规则、稳定分层和真实验证下持续补充可维护能力。

团队应持续坚持：

1. 需求输入结构化
2. 修改前读取上下文
3. page / flow / test 边界清晰
4. locator 使用真实 DOM 契约
5. 中文报告步骤可追踪
6. 验证结果来自真实命令

只要这些规则持续执行，AI 就能成为这个 POS 自动化项目的长期维护助手，而不是一次性脚本生成器。
