# 录制补充：部分子单支付与 Delivery Info 回归设计

## 目标

依据用户提供的真实 Playwright 录制，补齐以下两个点单页面回归场景：

1. `POS-16324`：订单平分为两个子单后，以现金支付其中一个子单，并在 Split 页面及 Recall 中校验已支付、未支付子单状态。
2. `POS-30575`：创建 Delivery 订单进入点单页后，打开客户 Info，校验姓名、电话、地址等预填信息与首次输入一致，同时保留保存后 Recall 客户信息回查。

实现完成并通过真实 UI 验证后，从待录制清单移除 `ORDER-PAGE-014`、`ORDER-PAGE-016`，覆盖矩阵更新为 23 条已覆盖、43 条需要录制。

## 方案选择

采用扩展现有 Page/Flow 并升级现有测试的方案。录制脚本只用于确定真实 DOM 契约，不直接复制动态订单 ID、`nth()`、深层 CSS 或逐键盘输入步骤。

未采用的方案：

- 直接复制录制脚本：实现快，但动态订单 ID、位置索引和页面文本组合会造成脆弱测试。
- 只通过 API 校验：无法证明 Split 子单状态和点单页 Info 预填信息真实展示。

## POS-16324 设计

### Page 与 Flow 边界

- `pages/split-order.page.ts`：集中维护真实 `payBtn-<index>` 子单支付入口；提供按子单序号打开 Payment 的页面动作；继续使用现有 `readSnapshot()` 读取子单号、菜品归属、金额和支付状态。
- `flows/split-order.flow.ts`：编排平分两份、选择一个子单支付、支付后返回 Split，以及提交分单结果。
- `flows/payment.flow.ts`：继续负责现金全额支付和支付完成态收敛，不复制支付细节。
- `pages/recall/recall-order-details.dialog.ts`：复用按母单号、子单号打开详情和读取支付状态的现有能力；如 Recall 卡片状态无法通过现有窄接口读取，仅新增基于真实卡片 DOM 的状态读取，不使用录制中的动态数据库 ID。

### 数据流与断言

1. 从 POS 首页走 Dine In 无桌路径进入点单页，添加稳定普通菜。
2. 从点单页进入 Split，平分为两个子单。
3. 记录运行期两个子单引用、金额和菜品归属；支付第一个子单。
4. 支付完成后回到 Split，断言第一个子单为 Paid，第二个子单为 Unpaid，两个子单仍保留平分菜品和金额守恒。
5. 提交并返回首页，从 Recall 精确打开母单，再分别打开两个运行期子单，断言已支付子单状态为 Success/Paid，未支付子单状态为 Unpaid。

测试不依赖录制中的 `#6-1`、`#6-2`、订单数据库 ID 或首卡位置。

## POS-30575 设计

### Page 与 Flow 边界

- `pages/delivery.page.ts`：继续负责首次填写电话、姓名、地址、房间号、邮编和备注并进入点单页。
- `pages/order-dishes/order-dishes-customer.section.ts`：扩展为点单页客户区域能力，集中维护客户按钮、Info 区域和预填字段的真实定位器；提供打开客户 Info、读取结构化客户信息和关闭弹层的方法。
- `flows/takeout.flow.ts`：增加 Delivery 信息录入后打开 Info 并返回快照的业务编排，避免测试包含逐控件操作。
- `tests/py-migrate/order.service.spec.ts`：升级现有 `POS-30575`，在保存前校验 Info 预填值，保存后继续按 Recall 校验客户信息。

### 数据流与断言

1. 使用 `test-data/order-service.ts` 的稳定 Delivery 输入进入点单页。
2. 点击运行期客户姓名/电话按钮打开 Info。
3. 读取姓名、规范化电话、地址、房间号、城市、州、邮编和备注中页面实际提供的字段。
4. 对电话做去格式化比较；地址按结构化字段或页面组合文本比较，不要求录制中的固定展示顺序。
5. 关闭 Info、添加菜品并保存；Recall 中继续断言客户姓名、电话、地址和备注与输入一致。

## 错误处理与隔离

- 页面方法等待真实加载信号，不使用 `waitForTimeout`。
- 所有输入后提交继续遵守至少 200ms 的输入稳定等待。
- 子单和订单全部使用运行期响应或页面快照标识。
- 若支付后页面返回点与录制不同，由 Flow 根据实际返回的 Split/Recall/Home 页面类型收敛，不把页面恢复策略隐藏在 Page 方法中。
- 测试不修改后台永久配置；若运行产生未支付订单，使用现有 Recall 清理能力在 `finally` 中清理可安全作废的测试订单。
