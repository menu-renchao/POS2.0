# 首批 112 个接口真实业务场景覆盖设计

## 背景

首批接口测试已经建立 Playwright + TypeScript 的 API 自动化框架，并登记了 112 个接口。当前执行口径是按业务场景聚合，一条 `test(...)` 内通过多个 `test.step(...)` 覆盖多个接口。这个结构利于复用前置数据，但顶层 case 数不会等于接口数。

下一阶段目标是把 112 个接口全部纳入真实业务场景验收：每个接口都必须能追踪到独立 `test.step(...)`、真实业务参数、前置数据来源、断言方式和当前执行状态。不能用错误参数或空对象把接口伪装成通过。

## 目标

1. 保持业务场景聚合，不强制一接口一条顶层 `test(...)`。
2. 每个接口至少对应一个独立中文 `test.step(...)`，step 标题包含方法和路径。
3. 每个接口在覆盖映射中有唯一记录，说明当前状态、所在 spec、业务链路、缺口和抓包需求。
4. 所有可稳定执行的接口都用专用测试环境真实数据跑通。
5. 外部依赖、后端未实现、缺少现场数据的接口必须明确标记，不能算作真实跑通。

## 非目标

1. 不把 112 个接口拆成 112 条顶层 `test(...)`。
2. 不为了凑通过率而使用无效 ID、空 body 或随机构造的业务参数。
3. 不面向共享环境运行破坏性写接口；本批接口测试默认面向专用测试环境执行写接口并负责清理。
4. 不在代码、文档或日志中固化真实 Cookie、会话 ID、支付凭据或会员敏感信息。

## 覆盖口径

每个接口的状态分为以下几类：

- `已真实跑通`：使用真实业务链路和有效参数执行，断言响应状态、信封结构和关键业务字段。
- `已纳入链路待增强`：接口已在业务 step 中执行，但断言或前置数据还不够完整。
- `需要现场抓包`：缺少真实 ID、请求体或业务前置，例如配送 `deliveryId`、真实会员、退款源支付单。
- `依赖外部服务`：接口需要第三方配送、支付、通知或其他外部状态。
- `后端未实现`：Swagger 或实测响应明确表示未实现。
- `仅契约烟测`：当前只能验证不返回 500，不能代表真实业务跑通。

验收时以覆盖映射表为准，而不是只看 Playwright 顶层 case 数。

## 业务链路设计

### 菜单配置链路

覆盖菜单、菜单组、分类、全局选项分类、全局选项和菜单全局搜索。链路先读取现有 POS 菜单，再在破坏性模式下创建短名称测试数据，执行查询、更新、批量或删除接口，最后按依赖倒序清理。

需要避免把后端未实现接口放进正向链路。当前 `/api/menu/group/listGroupCategoryItems` 已实测为未实现，保留缺口记录。

### 商品和 SPU 链路

覆盖商品查询、商品创建更新删除、快速编辑、商品选项、SPU 关联和库存操作。读取链路优先使用现有菜单分类下的真实商品；写链路创建受控测试商品，再执行后续更新、库存、SPU 映射和清理。

商品选项详情必须来自真实 `optionId`，如果当前样例没有返回选项，则列入现场抓包清单。

### 订单和支付链路

覆盖订单列表、订单明细、召回、保存、读取、支付、作废、退款、小费、拆单、合单、清台、重开和移动商品。订单类查询必须使用后端可解析的 `yyyy-MM-dd HH:mm:ss` 时间范围。

支付、退款、小费、作废等接口必须基于真实创建的订单和支付记录执行。不能用无效 ID 只验证不返回 500 来冒充真实业务覆盖。

### 后台配置链路

覆盖税费、折扣、角色等配置的列表、创建、删除。写接口使用短名称测试数据，并登记清理资源。

### 外部和缺口链路

配送跟踪、外部通知、第三方支付、后端未实现模块等接口不强行通过。每个接口在映射表中记录需要的现场动作，例如“创建一单外送并进入配送跟踪页，抓取带 `deliveryId` 的请求”。

## 数据和抓包规则

1. 现场抓包以浏览器 Network 中的真实请求为准，优先提供 curl。
2. 只需要提供目标接口的 URL、method、query 和 body；接口测试框架会固定注入测试环境后门 header，并自动通过 `/api/client/session/login` 和 `/api/login?passcode=11` 建立 API 登录态，不再通过本地 Cookie 环境变量传入鉴权信息。
3. 每次只收集一小批接口，建议 5 到 8 个，避免参数混淆。
4. 抓包后先把参数固化到 `test-data/api/` 或业务 helper，再把接口从缺口状态升级为真实业务 step。

## 断言规则

1. 通用成功接口校验：HTTP 状态不为 500，JSON 信封包含 `code`、`msg`，成功场景包含 `data`。
2. 业务负向接口校验：允许无 `data`，但必须校验 `code`、`msg` 和明确业务错误原因。
3. 写接口校验：创建后必须能通过读取或列表定位本次测试数据。
4. 清理校验：删除或作废后必须记录清理结果；清理失败不能吞掉，要在报告中暴露。
5. 外部依赖接口：如果无法稳定构造状态，只能标记为需要现场数据或外部依赖，不能记为真实跑通。

## 文档产物

1. `docs/api/112接口覆盖映射.md`：112 个接口逐条记录覆盖状态和缺口。
2. 后续每批升级后，更新映射表中的状态、spec、step 和抓包需求。
3. 首批抓包清单从映射表中筛选 `需要现场抓包`、`依赖外部服务`、`仅契约烟测` 的接口生成。

## 推进顺序

1. 生成 112 接口覆盖映射。
2. 先升级菜单、分类、商品、全局选项相关接口。
3. 再升级订单、支付、退款、小费、订单状态变更接口。
4. 再升级 SPU 库存和后台配置边界接口。
5. 最后处理配送跟踪、外部依赖和后端未实现接口。

## 验收标准

1. 覆盖映射中 112 个接口无遗漏。
2. 每个已真实跑通接口都能定位到 `test.step(...)`。
3. 专用测试环境中，非破坏性接口和写接口默认可执行；写接口必须登记清理并验证清理结果。
4. 所有未真实跑通接口都有明确原因和下一步需要的抓包动作。
5. 不提交真实 Cookie、真实会话 ID 或敏感支付/会员信息。
