# 点单页面提示词回归用例设计

## 目标

以 `C:\Users\nhqrt\Desktop\test_order_page_playwright_prompts.md` 中清理后的 66 条提示词为需求源，在现有 Playwright + TypeScript 项目中完成可维护的 UI 回归覆盖。已有用例只有在业务目标、关键前置、操作路径和核心断言均等价时才复用；无法从提示词、仓库代码和可运行页面确认的业务，统一转为结构化脚本录制需求，由录制结果补足真实 DOM 与业务路径后再实现。

## 当前基线

- 提示词：66 条，编号连续，无完全重复正文。
- `tests/py-migrate/order.service.spec.ts` 已包含点单、分单、客户信息、支付、小费和部分 option 场景。
- `tests/py-migrate/split-order-operation.spec.ts` 已包含大量分单、加收、退款、报表和支付相关回归能力。
- 按 Jira Key 的初步机械比对结果为：14 条全部命中、33 条未命中、19 条无 Jira Key。该结果只用于初筛，不能直接证明业务等价覆盖。
- 现有 `docs/plans/2026-05-20-order-page-py-migrate.md` 提供历史迁移分析；本设计以当前仓库和当前 66 条提示词为准，发现冲突时更新现状，不沿用过期结论。
- 现有 `docs/playwright-recordings-needed.md` 作为唯一录制需求清单，新增点单页面需求时按相同格式维护。

## 已选方案

采用“覆盖矩阵 + 缺口补齐”方案，不强制一条提示词对应一条新测试。

每条提示词必须在覆盖矩阵中有且只有一个最终状态：

1. `已等价覆盖`：现有测试完整覆盖业务目标、关键路径和核心断言，并记录对应 spec、测试标题和证据。
2. `待补断言`：现有测试覆盖主要路径，但缺少提示词要求的关键可观察结果。
3. `可直接实现`：仓库已有真实 Page/Flow API 和稳定定位契约，可以在不猜测 DOM 的前提下落地。
4. `需要录制`：缺少真实操作路径、稳定 DOM、配置入口或业务预期，不能安全实现。
5. `产品异常`：路径可以实现，但实际产品结果与提示词预期冲突；保留可执行证据并记录异常，不将失败伪装为通过。
6. `环境阻塞`：依赖当前环境不存在的账号、权限、数据、打印文件或外部服务；记录恢复执行所需的最小条件。

覆盖矩阵使用提示词编号作为稳定主键，Jira Key 只作为追踪字段。无 Jira Key 的场景仍必须按标题、步骤和断言完成语义映射。

## 交付结构

### 覆盖矩阵

新增 `docs/plans/2026-07-14-order-page-prompt-coverage.md`，逐条记录：

- 提示词编号、标题和 Jira Key；
- 目标业务域；
- 当前状态；
- 已有覆盖证据或计划落点；
- 缺少的 Page、Flow、测试数据或环境能力；
- 是否需要录制及录制需求编号；
- 最近验证命令和真实结果。

矩阵是 66 条需求的完成审计入口。测试代码、录制需求或产品异常状态发生变化时，同一批次同步更新矩阵。

### 测试文件

按业务域拆分，避免继续扩大热点 spec：

- 保留 `tests/py-migrate/order.service.spec.ts` 中已经稳定的基础点单场景；
- 分单相关优先复用或扩展 `tests/py-migrate/split-order-operation.spec.ts`，只有文件职责继续膨胀时才拆成更聚焦的 spec；
- 菜单/option、数量/合并菜、客户/权限、套餐/自定义类型等新增回归按共享前置和页面能力建立聚焦 spec；
- 不为了与提示词一一对应而复制已有测试。

所有 `describe`、`test`、`test.step` 标题均使用中文。测试元数据使用 Playwright 原生 `test(title, details, body)`，Jira Key 保留在 owning spec 的 `annotation` 中，业务标签遵守仓库 `AGENTS.md`。

### Page 与 Flow

- `pages/` 只新增真实页面结构、稳定定位器、单页动作和单页读取。
- `flows/` 只新增跨步骤业务编排、记录选择和业务策略。
- 热点门面 `pages/order-dishes.page.ts` 与 `pages/recall.page.ts` 保持薄委托，能力落到对应 section。
- 同一稳定元素只定义一个真实 DOM 契约；不使用 `.or()`、候选选择器遍历、多语言兜底或父页面回退猜测定位。
- 若页面缺稳定定位器，优先提出 `data-testid` 或录制需求，不使用脆弱 CSS/XPath 补洞。
- 输入后立即确认或提交并触发保存/API 的动作，确认前至少等待 200ms 的输入稳定时间。

### 测试数据与配置

- 稳定菜品、菜单组、分类、option、套餐、数量和预期金额进入 `test-data/`。
- 动态客户等数据使用 `test-data/` 工厂生成，spec 中不直接使用 `Date.now()`。
- 后台配置优先使用现有 API setup/fixture，并在场景结束后恢复；只有后台 UI 本身是被测行为或不存在 API 时才使用 UI 配置。
- POS 内页必须从 `myhome.html` 经 UI 进入，不使用 hash 或深链接。

## 分批实施

### 批次 0：66 条覆盖矩阵

先完成全部提示词的语义映射。对机械命中的 14 条逐条核对关键断言；对无 Jira Key 的 19 条根据业务步骤和历史迁移计划建立映射。该批次结束时，每条提示词必须具有状态和下一动作，不能留空白或模糊的“待确认”。

### 批次 1：现有能力可直接完成的场景

优先处理已有 Page/Flow 能力覆盖的分单、小费、Delivery 退出、数量读取等场景。每个新增行为遵循测试先行：先写能因能力缺失或断言缺失而失败的测试，再补最小 Page/Flow 实现。

### 批次 2：菜单、语言、搜索与 option

集中补菜单组/分类读取、语言切换、搜索结果和菜级 option 能力。类级 option 与菜级 option 不混用同一选择器；一级/二级 option 作为数据矩阵，但保留不同业务断言。

### 批次 3：数量、合并相同菜与 Modify

覆盖小数数量、指定价格、相同菜合并开关、Global Option add/count/reduce。配置变化通过 API fixture setup/teardown，UI 只验证 POS 行为。

### 批次 4：客户信息、权限、送厨与 Void

覆盖支付前客户信息、Guest Info、权限口令、Hold/Delay、送厨后删除和套餐子菜 note。该域涉及状态和权限弹窗，缺真实 DOM 时立即转录制需求，不猜测。

### 批次 5：套餐、自定义类型、打印、报表与礼品卡

覆盖套餐子菜 option/改价、自定义 Delivery、打印、Report Overview 和礼品卡。打印目录、云报表、礼品卡环境不可用时标记环境阻塞或需要录制，并写清恢复条件。

每个批次必须独立可审阅、可类型检查、可执行；不等待所有批次结束后才验证。

本设计是 66 条需求的总设计，不使用一个超大实施计划同时修改所有业务域。批次 0 单独形成覆盖矩阵计划；批次 1～5 在前一批证据稳定后分别形成实施计划。后续计划只能细化本设计，不能缩减 66 条总范围或把录制需求视为最终交付。

## 录制需求规则

只有满足以下任一条件时才创建录制需求：

- 当前代码与页面无法确定真实入口或操作顺序；
- 缺少稳定 DOM，无法判断应添加哪个 `data-testid`；
- 提示词描述与实际页面命名或状态转换不一致；
- 权限、Hold/Delay、套餐、打印或报表行为无法从现有封装推导；
- 最终断言所对应的数据位置或计算口径不明确。

每条录制需求必须包含：

1. 提示词编号、标题和 Jira Key；
2. 已知前置数据与账号/权限；
3. 从 POS 首页开始的录制范围；
4. 需要完整展示的点击、输入、弹窗、状态和最终断言位置；
5. 希望录制中保留的 DOM 信息，包括 `data-testid`、role、label、可见文本和关键网络请求；
6. 当前阻塞点以及录制返回后预计补充的 Page/Flow API。

录制完成后，将条目移动到“已收到，无需重复录制”，同时更新覆盖矩阵，不删除历史证据。

## 错误与异常处理

- 不用 `test.fail()` 掩盖尚未理解的业务；未理解场景进入录制需求。
- 已确认产品结果与需求冲突时，可以保留可执行预期失败覆盖，但必须写明需求预期、实际结果和复现证据。
- 环境 teardown、浏览器启动、账号或数据阻塞必须与业务断言失败分开记录。
- Recall 后续校验优先保存订单号等稳定字段精确定位；只能依赖最新订单时，在矩阵中记录串行执行和并发干扰风险。
- 不使用 `waitForTimeout`、`expect.poll()` 或 `expect(...).toPass()` 掩盖时序问题；需要轮询时使用 `utils/wait.ts` 的 `waitUntil()`，只断言最终稳定结果。

## 测试与验证

每个实现任务至少执行：

1. 新测试的 RED 运行，确认因目标能力缺失而失败；
2. 最小实现后的目标测试 GREEN 运行；
3. 受影响 spec 或同域回归；
4. `npx tsc --noEmit`；
5. 定位器与架构规则静态检查，包括 `waitForTimeout`、直跳内页、散落 locator 和非中文步骤；
6. 覆盖矩阵状态与真实命令结果同步。

若 live POS 环境无法运行，类型检查和静态检查不能替代业务通过；矩阵必须标记“环境阻塞”，并保留待执行命令。

## 完成标准

只有同时满足以下条件时，66 条提示词迁移才算完成：

- 覆盖矩阵包含连续的 1～66，且没有空状态；
- 每条为 `已等价覆盖`、已实现并验证，或经可重复执行证据确认的产品异常；
- 不存在未解除的 `环境阻塞`；若外部环境长期不可用，只有金将军明确接受该项延期后，才允许从本目标范围中单独拆出，不能由实现者自行视为完成；
- 所有 `需要录制` 条目均已收到录制并完成实现，不能以录制需求本身替代最终覆盖；
- 每条已有覆盖都能定位到具体测试标题和关键断言；
- 新增 Page/Flow 方法使用中文 `@step(...)`；
- 所有适用测试已运行并记录真实结果，TypeScript 类型检查通过；
- 录制需求、覆盖矩阵和测试代码三者状态一致；
- 不存在因本次迁移新增的重复用例或猜测型选择器。

## 非目标

- 不在本任务中重构无关 API 测试或 Jenkins 配置。
- 不为了提高数量而添加只验证页面可见性的弱回归。
- 不将 Python 源实现逐行翻译成 Playwright 点击脚本。
- 不假设所有 Jira Key 相同的用例都等价，也不因标题相似而删除不同边界覆盖。
