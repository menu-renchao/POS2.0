# 输入提交前稳定等待设计

## 目标

为 POS UI 自动化项目建立统一约束：任何涉及 `input` 编辑并紧接确认/提交、且该确认动作会触发接口或提交修改的页面操作，在点击确认按钮前必须等待至少 `200ms`，以等待输入框稳定。

## 问题背景

当前多个页面对象在 `fill()` 后立即点击确认或提交按钮。真实页面存在输入法、虚拟键盘、受控组件状态同步、延迟写回等行为，导致自动化可能在输入值尚未稳定落地时就触发请求，形成偶发错误。

这个问题已在库存沽清设置链路暴露，但代码检索显示相同模式也存在于点单页数量/重量/价格输入、Recall 手动搜索与 Void Note 提交、分单输入确认等路径。

## 设计原则

1. 规则统一收口在 `pages/`，不让 `flows/` 或 spec 手工插入等待。
2. 只处理“输入后提交”的动作，不污染纯导航或无输入提交按钮。
3. 同一页面动作里，如果有多个输入框，最后一次输入完成后统一等待 `200ms` 再提交。
4. 使用共享 helper，避免各页面散落魔法数字。
5. 不使用 `waitForTimeout`；稳定等待 helper 自身负责封装最小固定等待。

## 影响范围

首批接入以下页面对象中的输入后提交动作：

- `pages/inventory-stock-setting.page.ts`
- `pages/order-dishes.page.ts`
- `pages/recall.page.ts`
- `pages/split-order.page.ts`

接入点包括但不限于：

- 沽清/有限库存数量输入后点击 `Confirm`
- 数量、重量、价格、自定义加收等输入后点击 `Confirm`
- Recall 手动搜索输入后点击 `Search`
- Recall Void Note 编辑与恢复库存切换完成后点击 `Void`
- 分单输入后点击 `Confirm`

## 实现方案

### 共享 helper

在 `utils/` 新增一个小型 helper，例如 `waitForInputSettled()`，默认等待 `200ms`。

职责仅包括：

- 暴露统一常量
- 在最后一次输入后执行最小稳定等待

不负责：

- 自动点击按钮
- 条件轮询
- 输入本身

### 页面对象接入

页面对象内部在“最后一次输入完成后、点击确认前”显式调用 helper。

这样可以保持每个页面动作的语义清晰，例如：

- `setLimitedStockQuantity()` 负责输入
- `saveInventoryConfig()` 在真正提交前执行稳定等待

对于同一个页面对象中“输入”和“确认”拆分为两个方法的情况，采用“在确认方法里统一等待”的策略，确保调用方不需要记忆规则。

## 测试策略

采用 TDD：

1. 先补库存设置回归测试，构造“200ms 内提交会读到旧值”的最小场景。
2. 再补一个跨页面代表测试，优先选择 Recall 手动搜索或分单输入确认。
3. 回跑受影响的现有契约测试集，确认统一 helper 不引入回归。

## 文档规则更新

在 `AGENTS.md` 新增自动化规则：

- 任何涉及 `input` 编辑并紧接确认/提交、且该动作会触发接口或提交修改的页面操作，在点击确认按钮前必须等待至少 `200ms`，以等待输入框稳定。

## 风险与边界

- 该规则不会用于纯键盘输入后本地 UI 切换、不触发接口的场景。
- 该规则不会替代已有的页面加载、按钮启用、接口完成等待。
- 如果个别页面未来需要比 `200ms` 更长的稳定时间，应在页面对象内部局部覆盖，但默认仍统一使用共享 helper。
